DNS: nombres en vez de números

Nadie quiere memorizar 192.168.4.101. El DNS traduce nombres como www.miempresa.net a direcciones IP y al revés. En este tema montas tu propio servidor DNS con bind9 sobre Ubuntu y creas tu zona.

1. Qué resuelve el DNS

Analogía

La agenda del móvil. Tú marcas "Mamá", no su número. El DNS es esa agenda a escala de Internet: guarda la correspondencia nombre → número para que las personas usemos nombres y las máquinas sigan usando IPs. Cuando añades un contacto nuevo, lo das de alta una vez y todo el mundo puede llamarlo por su nombre.

El proceso es jerárquico y se hace por delegación: tu equipo pregunta a su servidor DNS; si este no conoce el nombre, pregunta hacia arriba (raíz → .net → miempresa.net) hasta encontrar al servidor autoritario de esa zona, que responde con la IP. El resultado se guarda un tiempo en caché (TTL) para no repetir el viaje.

Diagrama: una consulta DNS

topologia — resolución DNS
  cliente (u20zerbitzuak)        servidor DNS (u18zerbitzaria)
        │  "¿IP de www.josebabilbao.net?"   │
        │ ─────────────────────────────────▶│  busca en su zona
        │                                   │  www → CNAME → A → 192.168.4.101
        │ ◀─────────────────────────────────│  "es 192.168.4.101"
        ▼
   guarda en caché (TTL) y conecta

2. Registros de recursos (RR)

Una zona DNS es un fichero de texto lleno de registros. Cada registro tiene la forma Propietario · TTL · Clase · Tipo · RDATA. La clase casi siempre es IN (Internet). Los tipos que usarás:

TipoPara quéEjemplo (RDATA)
SOAInicio de la zona: servidor autoritario y parámetrosserial, refresh, retry…
NSServidores de nombres de la zonau18zerbitzaria.
ANombre → dirección IPv4192.168.4.111
PTRIP → nombre (resolución inversa)w10zer.josebabilbao.net.
CNAMEAlias de otro nombrewww → u18zerbitzaria
MXServidor de correo del dominio10 u18zerbitzaria
TXTTexto libre (verificaciones, SPF…)"v=spf1 …"
Nota · ¿por qué dos zonas?

La zona directa resuelve nombre → IP (registros A). La zona inversa resuelve IP → nombre (registros PTR) y vive en un dominio especial in-addr.arpa con los octetos al revés. Son ficheros distintos porque responden preguntas opuestas.

3. Instalación paso a paso (Ubuntu · bind9)

Trabajamos sobre tres VMs del Tema 0: u18zerbitzaria será el servidor DNS (ejemplo 192.168.4.101); w10zerbitzuak (192.168.4.111) y u20zerbitzuak (192.168.4.112) serán los clientes. Cambia el nombre de dominio por el tuyo (nombreapellido.net).

3.1 · Instalar bind9 en el servidor

u18zerbitzaria
$ sudo apt update && sudo apt install -y bind9 bind9utils dnsutils

Después, en las conexiones de red del servidor, pon como DNS su propia IP (192.168.4.101): se resolverá a sí mismo.

3.2 · Declarar la zona directa

Edita /etc/bind/named.conf.local y declara tu zona apuntando a su fichero de datos.

u18zerbitzaria
$ sudo nano /etc/bind/named.conf.local
/etc/bind/named.conf.local
zone "josebabilbao.net" {
    type master;
    file "/etc/bind/db.josebabilbao";
};

3.3 · Crear y editar el fichero de zona

Copia la plantilla db.local y edítala con tus datos.

u18zerbitzaria
$ cd /etc/bind
$ sudo cp db.local db.josebabilbao
$ sudo nano db.josebabilbao

Dentro, sustituye localhost por tu dominio, 127.0.0.1 por la IP del servidor y añade un registro A por cada cliente:

/etc/bind/db.josebabilbao
$TTL    604800
@       IN      SOA     josebabilbao.net. admin.josebabilbao.net. (
                          2         ; Serial (súbelo en cada cambio)
                     604800         ; Refresh
                      86400         ; Retry
                    2419200         ; Expire
                     604800 )       ; Negative Cache TTL
@       IN      NS      josebabilbao.net.
@       IN      A       192.168.4.101
w10zer  IN      A       192.168.4.111
u20zer  IN      A       192.168.4.112
Error común

Olvidar el punto final en los nombres absolutos (josebabilbao.net.) o no subir el Serial tras editar. Sin subir el serial, los clientes seguirán viendo la versión antigua en caché.

3.4 · Reiniciar y comprobar

u18zerbitzaria
$ sudo named-checkzone josebabilbao.net /etc/bind/db.josebabilbao
$ sudo systemctl restart bind9
$ sudo systemctl status bind9
# probar la resolución contra tu propio servidor
$ host josebabilbao.net 192.168.4.101
$ host w10zer.josebabilbao.net 192.168.4.101
Nota · nombre del servicio

En Ubuntu 18/20 el servicio es bind9 (también vale sudo service bind9 restart). En 22.04+ la unidad puede llamarse named: usa sudo systemctl restart named si bind9 no existe.

3.5 · Zona inversa (IP → nombre)

Añade una segunda zona en named.conf.local para la red 192:

/etc/bind/named.conf.local
zone "192.in-addr.arpa" {
    type master;
    file "/etc/bind/db.192";
};
u18zerbitzaria
$ sudo cp /etc/bind/db.127 /etc/bind/db.192
$ sudo nano /etc/bind/db.192

Sustituye localhost por tu dominio y crea los PTR con los octetos al revés (sin el 192 inicial, que ya está en el nombre de la zona). Para 192.168.4.111 el propietario es 111.4.168:

/etc/bind/db.192
$TTL    604800
@       IN      SOA     josebabilbao.net. admin.josebabilbao.net. ( 2 604800 86400 2419200 604800 )
@       IN      NS      josebabilbao.net.
101.4.168   IN  PTR     josebabilbao.net.
111.4.168   IN  PTR     w10zer.josebabilbao.net.
112.4.168   IN  PTR     u20zer.josebabilbao.net.
u18zerbitzaria
$ sudo systemctl restart bind9
$ host 192.168.4.111 192.168.4.101   # debe devolver w10zer.josebabilbao.net

3.6 · Apuntar los clientes al servidor

En u20zerbitzuak y w10zerbitzuak, pon como DNS primario la IP del servidor (192.168.4.101) y como secundario 8.8.8.8. Comprueba:

u20zerbitzuak — cliente
$ ping -c2 u20zer.josebabilbao.net
$ host josebabilbao.net
Si el ping falla

Quita temporalmente el DNS secundario 8.8.8.8 para forzar que pregunte solo a tu servidor; vuélvelo a poner luego. En el cliente Windows, desactiva el cortafuegos si bloquea las respuestas.

4. Ejercicios

Nivel 1 · Reproducir Resolver en el simulador ›

Con la zona por defecto del simulador, consulta w10zer.josebabilbao.net, www.josebabilbao.net y la IP 192.168.4.112.

Cómo sabes que está bien: www llega a la IP del servidor pasando por un CNAME, y la consulta inversa de .112 devuelve u20zer.

Nivel 2 · Modificar Añadir un host ›

En el simulador añade u22 IN A 192.168.4.113 y un alias intranet IN CNAME u22. Resuelve intranet.josebabilbao.net.

Cómo sabes que está bien: intranet → u22 → 192.168.4.113. Luego replica ese cambio en tu db real y reinicia bind9.

Nivel 3 · Crear Tu zona completa (3_AS) ›

Sobre u18zerbitzaria, monta bind9 con tu dominio nombreapellido.net, zona directa con los tres equipos y zona inversa funcionando.

Cómo sabes que está bien: desde un cliente, host nombreapellido.net y host w10zer.nombreapellido.net resuelven, y host 192.168.4.111 devuelve el nombre.

Nivel 4 · Desafío Reenviadores y MX ›

Configura un forwarder en named.conf.options para que tu DNS resuelva también nombres de Internet a través de 8.8.8.8, y añade un registro MX para tu dominio. Sin solución proporcionada.

5. Autoevaluación

Comprueba el Tema 2

—

¿Qué tipo de registro asocia un nombre con una IPv4?

A (Address) = nombre → IPv4. El PTR hace lo inverso, IP → nombre.

¿En qué fichero declaras tu zona josebabilbao.net?

En named.conf.local se declaran las zonas; los datos van en el fichero db.* que indica esa declaración.

Para la IP 192.168.4.111 en la zona inversa 192.in-addr.arpa, ¿cuál es el propietario del PTR?

Se escriben los octetos al revés y se omite el 192 (ya está en el nombre de la zona): 111.4.168.

Editas la zona pero los clientes siguen viendo datos viejos. ¿Qué olvidaste?

El Serial avisa de que la zona cambió; sin subirlo (y reiniciar) se sirve la versión cacheada.
